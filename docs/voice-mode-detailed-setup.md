# Voice Mode — Complete Working Setup (April 2026)

This document captures the exact working configuration of N4's voice mode as of April 10, 2026. If any future change breaks voice mode, restore the code to match what is documented here.

---

## Architecture Overview

Voice mode uses the **OpenAI Realtime API** with **WebRTC** for live two-way audio. The flow is:

```
Browser (WebRTC + microphone)
    ↓ SDP offer (via JSON POST)
Supabase Edge Function: realtime-conversation
    ↓ FormData POST (sdp + session config)
OpenAI /v1/realtime/calls
    ↓ SDP answer (plain text)
Edge function wraps answer in JSON → returns to browser
    ↓
Browser sets remote SDP → WebRTC audio stream established
    ↓
Data channel receives events (transcription, audio, tool calls)
```

## Critical: OpenAI /v1/realtime/calls Request Format

The single most important thing to get right. This has broken voice mode multiple times.

### What Works (current, verified working)

The edge function uses **native `FormData`** with **NO manual Content-Type header**. The `fetch` API auto-generates the correct multipart boundary.

```typescript
const fd = new FormData();
fd.set("sdp", sdp);
fd.set("session", JSON.stringify(config));

return await fetch("https://api.openai.com/v1/realtime/calls", {
  method: "POST",
  headers: {
    Authorization: `Bearer ${OPENAI_API_KEY}`,
  },
  body: fd,
});
```

### What Does NOT Work

- Manually constructing multipart body strings with custom boundaries
- Setting `Content-Type: multipart/form-data; boundary=...` manually
- Using `application/sdp` content type on the SDP form field
- Using `application/json` content type on the session form field

These all cause 502/504 errors or Cloudflare timeout pages from OpenAI.

### Why

The official OpenAI Node.js example (https://developers.openai.com/api/docs/guides/realtime-webrtc) uses plain `FormData` with `fd.set()` and does NOT set Content-Type. The server expects standard multipart/form-data encoding as generated by the browser/runtime FormData implementation.

## Session Config (Exact Working Version)

```typescript
const initialSessionConfig = {
  type: "realtime",
  model: "gpt-realtime",
  instructions: fullInstructions,
  output_modalities: ["audio"],
  audio: {
    input: {
      transcription: { model: "gpt-4o-transcribe" },
      turn_detection: { type: "semantic_vad" },
    },
    output: { voice: "marin" },
  },
};

// Tools added conditionally:
if (realtimeTools.length > 0) {
  initialSessionConfig.tools = realtimeTools;
  initialSessionConfig.tool_choice = "auto";
}
```

### Config Rules

| Field | Value | Notes |
|---|---|---|
| `model` | `"gpt-realtime"` | NOT `"gpt-realtime-1.5"` or any versioned string. The alias auto-resolves. |
| `output_modalities` | `["audio"]` | NOT `["audio", "text"]`. Text modality is not needed for voice-only mode. |
| `turn_detection.type` | `"semantic_vad"` | NOT `"server_vad"`. Semantic VAD handles conversational pauses naturally. No manual threshold/padding params needed. |
| `transcription.model` | `"gpt-4o-transcribe"` | For input speech-to-text. |
| `voice` | `"marin"` | The Sica persona voice. |

## Retry Strategy (3 attempts)

The edge function tries 3 times with decreasing complexity:

1. **Attempt 1 (full)**: All tools + full instructions + news context
2. **Attempt 2 (reduced)**: Max 8 tools + instructions capped at 3000 chars
3. **Attempt 3 (minimal)**: No tools, no instructions — just `{ type: "realtime", model: "gpt-realtime", audio: { output: { voice: "marin" } } }`

Each attempt has a 30-second timeout. This ensures voice mode at least connects even if MCP servers are slow.

---

## Timeout Budget

| Phase | Timeout | Notes |
|---|---|---|
| Individual MCP server fetch | 5000ms | Per-server timeout for initialize + tools/list |
| MCP tool fetch phase (all servers combined) | 8000ms | If this times out, session proceeds with zero MCP tools |
| OpenAI /v1/realtime/calls attempt | 30000ms | Per attempt; 3 attempts total |
| Supabase Edge Function total | ~60s | Deno runtime limit |

The MCP phase timeout (8s) is deliberately lower than the OpenAI timeout (30s) to leave enough headroom for the actual API call.

## Client-Side WebRTC Setup (`realtimeConversation.ts`)

### Connection Flow

1. Request microphone with noise suppression, echo cancellation, and auto gain
2. Create `RTCPeerConnection` with Google STUN servers
3. Create `HTMLAudioElement` with autoplay for AI audio output
4. Add microphone tracks to peer connection
5. Create data channel named `"oai-events"` (ordered)
6. Create SDP offer, set as local description
7. POST offer to `realtime-conversation` edge function
8. Receive SDP answer, set as remote description
9. WebRTC connection established — audio flows both ways

### Data Channel Events Handled

| Event Type | Action |
|---|---|
| `input_audio_buffer.speech_started` | Status -> listening |
| `input_audio_buffer.speech_stopped` | (no action) |
| `conversation.item.input_audio_transcription.completed` | Fire onTranscription handler |
| `response.audio_transcript.delta` / `response.output_audio_transcript.delta` | Accumulate response text |
| `response.audio_transcript.done` / `response.output_audio_transcript.done` | Fire onResponseText (final) |
| `response.audio.delta` / `response.output_audio.delta` | Status -> speaking |
| `response.output_item.added` (function_call) | Status -> tool_calling, fire onToolCall |
| `response.function_call_arguments.delta` | Accumulate tool arguments |
| `response.function_call_arguments.done` | Finalize tool arguments |
| `response.output_item.done` (function_call) | Queue completed tool call |
| `response.done` | Execute queued tool calls or status -> active |

### Tool Call Execution

When a tool call completes:
1. If tool is in CLIENT_TOOLS set (change_symbol, switch_workspace, etc.), execute locally via `onClientToolCall` handler
2. Otherwise, POST to `realtime-tool-call` edge function which forwards to the appropriate MCP server
3. Send result back via data channel as `conversation.item.create` with `function_call_output`
4. Send `response.create` to resume AI response generation

### Client Tools (executed in browser, not via MCP)

```
change_symbol, change_timeframe, change_chart_type, toggle_indicator,
add_to_watchlist, remove_from_watchlist, create_watchlist, switch_watchlist,
switch_right_panel, switch_left_tab, add_to_ticker, remove_from_ticker,
add_clock, remove_clock, switch_workspace
```

## Edge Function: Tool Format Conversion

MCP tools are converted to OpenAI Realtime format:

```typescript
// MCP format -> OpenAI Realtime format
{
  type: "function",
  name: tool.name,
  description: tool.description.slice(0, 200),  // Capped at 200 chars
  parameters: {
    type: schema.type || "object",
    properties: schema.properties,
    required: schema.required,
  },
}
```

Maximum 20 tools sent to OpenAI (hard cap to avoid payload size issues).

The toggle function in MarketsDashboard.tsx:

1. If conversation active: calls `stopConversationSession()` and sets status to `idle`
2. If not active:
   - Collects last 6 chat messages as conversation context
   - Gets/refreshes Supabase auth token
   - Fetches Smithery service token for MCP connections (if any Smithery connections exist)
   - Builds MCP server list: global-monitor + user's Smithery connections
   - Calls `startConversationSession()` with all handlers and options
   - Handlers wire transcriptions and responses into the AI chat message list

## Common Failure Modes and Fixes

### 502/504 with Cloudflare HTML in error body
**Cause**: OpenAI request timing out or malformed multipart body.
**Fix**: Ensure native `FormData` is used (not manual multipart string). Ensure MCP tool fetch timeout is low enough (8s) to leave time for the OpenAI call.

### Wrong voice / hallucinated speech / not following instructions
**Cause**: Wrong model identifier (e.g., `gpt-realtime-1.5` instead of `gpt-realtime`) or wrong turn detection type (`server_vad` instead of `semantic_vad`).
**Fix**: Use exactly `model: "gpt-realtime"` and `turn_detection: { type: "semantic_vad" }`.

### Voice connects but no tool calls work
**Cause**: Tool definitions malformed or tools array exceeds 20 items.
**Fix**: Verify `convertMCPToolsToRealtimeFormat()` output. Check tool count cap.

### Microphone access denied
**Cause**: Browser permissions or HTTPS requirement.
**Fix**: App must be served over HTTPS (or localhost). User must grant microphone permission.
